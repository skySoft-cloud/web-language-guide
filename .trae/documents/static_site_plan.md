# 静态网站建设计划

## 一、项目调研结论

### 现有资源
- **项目名称**：web-language-guide
- **已有文件**：
  - `README.md` / `README.en.md` - 项目介绍文档
  - `多语言规范.md` - 多语言使用规范内容源文件
  - `多语言规范.html` - 已有精美单页HTML，包含完整的样式和交互（侧边栏导航、锚点追踪、回到顶部、响应式）

### 现有设计风格分析
- **配色方案**：蓝紫色渐变主题（#2563eb → #7c3aed）
- **布局**：固定侧边栏 + 主内容区
- **组件**：卡片式section、表格、代码块、FAQ折叠、流程步骤、标签徽章
- **交互**：滚动导航高亮、回到顶部按钮、移动端侧边栏切换

### 用户需求
1. 创建一个**静态网站**，可发布到托管平台
2. 将现有多语言规范页面作为**一个模块**，未来可扩展其他规范模块
3. **风格统一美观**，保持现有设计水准
4. **更新机制**：通过更新Markdown文件，由AI调整对应HTML文件

## 三、实施步骤

### 步骤 1：创建全局样式和脚本（统一风格）

**文件**：`assets/css/style.css`、`assets/js/main.js`

**内容**：
- 提取现有HTML中的CSS样式，形成统一主题
- 定义设计变量（颜色、字体、间距、阴影等）
- 封装通用组件样式（卡片、按钮、表格、代码块、标签等）
- 实现通用交互（侧边栏导航、锚点追踪、回到顶部）

### 步骤 2：创建首页（导航入口）

**文件**：`index.html`

**内容**：
- 网站标题和简介
- 规范模块卡片列表（展示各个规范模块）
- 快速入口链接
- 响应式设计，适配移动端

### 步骤 3：创建规范列表页

**文件**：`specs/index.html`

**内容**：
- 所有规范模块的完整列表
- 每个模块的简介、更新时间、标签分类
- 搜索/筛选功能（可选）

### 步骤 4：优化现有多语言规范页面

**文件**：`specs/i18n/index.html`

**内容**：
- 将内联CSS替换为引用全局样式文件
- 将内联JS替换为引用全局脚本文件
- 保持原有内容和交互功能不变
- 统一导航结构与首页风格一致

### 步骤 5：建立Markdown到HTML的更新机制

**约定**：
- 每个规范模块对应一个目录（如 `specs/i18n/`）
- 目录内同时存放 `.md`（内容源）和 `.html`（页面）
- 内容更新流程：编辑 `.md` 文件 → 通知AI更新对应 `.html` 文件

## 四、设计风格要点

### 配色方案（保持一致）
| 用途 | 颜色 |
|---|---|
| 主色调 | `#2563eb` → `#7c3aed`（渐变） |
| 背景 | `#f8f9fb` |
| 卡片 | `#fff` |
| 边框 | `#e8ecf1` |
| 文字 | `#1d2939`（主）、`#475467`（次）、`#98a2b3`（淡） |

### 布局规范
- 侧边栏宽度：260px
- 内容区间距：44px 56px
- 卡片圆角：10px
- 卡片阴影：`0 1px 3px rgba(0,0,0,.06)`

### 字体规范
- 主字体：`-apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif`
- 代码字体：`"SF Mono", "Fira Code", "Cascadia Code", Consolas, monospace`

## 五、潜在依赖与考虑

### 依赖
- **无外部框架依赖**：纯HTML/CSS/JavaScript，确保可静态托管
- **无构建工具**：无需Webpack/Vite，直接发布即可

### 托管平台兼容性
- 支持GitHub Pages、Gitee Pages、Netlify、Vercel等静态托管
- 目录结构支持路由（`specs/i18n/index.html` 可通过 `specs/i18n/` 访问）

### 扩展考虑
- 新增规范模块只需在 `specs/` 下创建新目录
- 首页和列表页自动包含新模块（通过导航配置）

## 六、风险处理

### 风险 1：样式冲突
- **应对**：使用CSS命名空间，避免全局污染
- **措施**：所有样式添加 `.site-` 前缀或使用BEM命名规范

### 风险 2：Markdown格式差异
- **应对**：建立标准Markdown格式规范
- **措施**：规定标题层级、表格格式、代码块标记等

### 风险 3：导航维护成本
- **应对**：将导航配置抽离为独立JSON文件
- **措施**：新增模块时只需更新配置文件，无需修改多个HTML

## 七、验证标准

1. **页面可访问性**：所有页面均可直接通过浏览器打开
2. **响应式**：在桌面端和移动端均有良好表现
3. **风格统一**：所有页面使用相同的配色、字体、组件样式
4. **交互完整**：侧边栏导航、锚点追踪、回到顶部功能正常
5. **静态托管兼容**：无需服务器端支持，纯静态文件可直接部署
